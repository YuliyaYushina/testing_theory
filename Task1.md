# **План тестирования**
## **Проект: Задание 1: тестирование API: Получение статуса продукта**
## **Версия: <1.0>**

### **1. Введение**

##### **1.1. Цель**

Данный документ описывает стратегию тестирования реализованного REST API GET /products/{productId}/status?authToken=&recalculate=&owner=&region=.
**Цель** - проверка корректности возвращаемых статусов продукта, работы системы авторизации и обработки ошибок.

##### **1.2. Обоснование**

Метод REST API GET /products/{productId}/status?authToken=&recalculate=&owner=&region= позволяет запрашивать статус продукта (1 - Готов/0 - Не готов). В качестве входных данных метод может принимать параметры recalculate, owner, region.

##### **1.3. Область тестирования**

**Этапы тестирования:** функциональное тестирование API, тестирование безопасности, негативное тестирование.
**Что тестируем:** получение статуса продукта, авторизация, обработка ошибок.
**Что НЕ тестируем:** UI-тестирование, тестирование производительности (Load/Stress).
**Риски:** неверная обработка "null", обработка кириллицы.

### **2. Требования к тестированию**

- Проверка корректности авторизации (16-значный authToken).
- Проверка логики фильтрации (recalculate, owner, region).
- Проверка кодов ответа HTTP: 200, 401, 404, 500.
- Валидация JSON-структуры ответа.

### **3. Стратегии тестирования**

##### **3.1. Типы тестирования**

##### **3.1.2. Функциональное тестирование**

**Объект:** Проверка корректности полученного статуса продукта.
**Техника:** Таблица принятий решений, попарное тестирование. Позитивные и негативные сценарии (при отсутствии продукта, ошибка базы данных или сервера).
**Критерий завершения:** 100% критических сценариев пройдены успешно.

##### **3.1.9. Тестирование безопасности и контроля доступа**

**Объект:** Проверка authToken - набор из 16 символов, включающие цифры и буквы.
**Техника:** Граничные значения для authToken (16 символов (цифры+буквы), 15 символов, 17 символов, пусто, спецсимволы)
**Критерий завершения:** при корректном токене - получение статуса продукта, при неверном или отсутствующем токене - ответ с кодом 401.

#### **Тест-кейс №2: Запрос несуществующего продукта**

**Описание:** Проверка получения статуса несуществующего продукта

**Шаги:**
1. Сформировать GET запрос с productId = 999999 (заведомо несуществующий ID).
2. Указать валидный authToken.
3. Отправить запрос.

**Ожидаемый результат:** 
HTTP-код: 404 Not Found.
Тело ответа пустое или содержит сообщение о том, что продукт не найден.

#### **Тест-кейс №3: Проверка авторизации (authToken)**

**Описание:** Проверка доступа при вводе разных токенов.

**Шаги:**
1. Сформировать GET запрос: /products/{productId}/status с существующим productId.
2. Добавить параметр authToken (См. ожидаемый результат).
3. Отправить запрос.

**Ожидаемый результат:**

| Вводимое значение authToken | Ожидаемый код | Ожидаемый результат |
|-|-|-|
| A1B2C3D4E5F6G7H8 | 200 OK | Успешный ответ со статусом |
| A1B2C3D4E5F6G7H9 | 401 Unauthorized | Доступ запрещен: Ошибка авторизации при использовании валидного токена |
| Оставить пустым | 401 Unauthorized | Доступ запрещен: токен отсутствует |
| A1B2C3D4E5F6G7H (15 символов) | 401 Unauthorized | Ошибка: токен неверной длины |
| A1B2C3D4E5F6G7H34 (17 символов) | 401 Unauthorized | Ошибка: токен неверной длины |
| ABC!@#1234567890 | 401 Unauthorized | Ошибка: неверный формат токена |


#### **Тест-кейс №4: Обработка внутренней ошибки сервера (500)**

**Описание:** Проверка формата ответа при возникновении системного сбоя.

**Шаги:**
1. Сформировать GET запрос: /products/{productId}/status.
2. Добавить параметр authToken (A1B2C3D4E5F6G7H8).
3. Спровоцировать ошибку сервера (например, передать в productId значение, вызывающее сбой базы данных, или дождаться планового отключения сервиса).
4. Отправить запрос.

**Ожидаемый результат:**
HTTP-код: 500 Internal Server Error.
Тело ответа в формате JSON содержит поле: {"errorMessage": "Текст описания ошибки"}.
